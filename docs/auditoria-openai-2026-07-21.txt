1. Problemas principales
1.1 Respuestas estructuradas incompatibles o mal parseadas

Aparecen tres síntomas:
Respuesta con formato inesperado: 'content'
Extra data: line 123 column 1
Extra data: line 81 column 1
Esto sugiere al menos dos problemas:

   1. El adaptador espera una estructura de respuesta distinta a la enviada por
      algún proveedor.
   2. El modelo devuelve varios objetos JSON, texto adicional o bloques
      Markdown después del JSON esperado.

El primer error puede venir de mezclar formatos como:
response["choices"][0]["message"]["content"]
con APIs que devuelven algo como:
response["output"][...]
o donde contentes una lista de bloques en vez de una cadena.

Mejora recomendada

Cree una interfaz normalizada por proveedor:
class LLMResponse: text: str finish_reason: str | None usage: Usage
provider: str model: str raw_response_id: str | None
Cada adaptador —OpenAI, ZAI, etc.— debe transformar su respuesta nativa a este
formato antes de pasarla al generador.
Para la salida de ideas:

   •   Usar esquema JSON o salida estructurada nativa cuando el proveedor lo
       permita.
   •   Validar con Pydantic.
   •   Pedir un objeto de raíz inequívoco:

{ "ideas": [ { "title": "...", "description": "...", "features": [] }
] }

   •   Prohibir expresamente Markdown y texto posterior.
   •   Implementar una única reparación controlada antes de descartar la
       respuesta.
   •   Registrar qué proveedor y modelo produjo el JSON inválido.

El salvamento actual es útil, pero no debería ser el flujo normal.
1.2 La población inicial queda incompleta

El motor solicita ocho individuos:
population_generated generated=5 requested=8
Esto puede degradarse considerablemente con un algoritmo evolutivo:

   •   Menos diversidad inicial.
   •   Menos opciones de selección.
   •   Mayor riesgo de convergencia prematura.
   •   Métricas generacionales sin comparables.
   •   Operadores de cruce seleccionando repetidamente los mismos padres.

Además, aparece más adelante:
population_generated generated=3 requested=1
Aquí ocurre lo contrario: se piden 1 y se aceptan 3.

Mejora recomendada

Aplicar un contrato estricto de cardinalidad:
accepted = valid_unique_ideas[:requested] missing = requested -
len(accepted)
Si faltan individuos:

   1. Reintentar únicamente missing.
   2. Usar lotes pequeños.
   3. Cambiar de proveedor si el primero tiene mala tasa de validez.
   4. No comenzará la generación hasta alcanzar un mínimo configurable.

Por ejemplo:
min_population_ratio = 0.875
Para pop_size=8, se podría exigir al menos 7, aunque lo ideal es completar los 8.
También conviene registrarse:
population_request requested=8 population_response raw_items=7 valid=5
duplicate=1 invalid=1 accepted=5 population_refill requested=3
accepted=3 final_size=8
Ahora mismo no es posible saber si las ideas se perdieron por parsing, validación,
duplicados o límites de salida.

1.3 Gestión ineficiente de límites tarifarios

OpenAI devuelve repetidamente:
Rate limit excedido en openai
Sin embargo, el enrutador vuelve a elegir OpenAI en operaciones posteriores y vuelve a
fallar. Esto indica que el failover es por petición, pero no existe —o no funciona— un
estado global de salud del proveedor.
Mejora recomendada: disyuntor

Cuando un proveedor devuelve 429o varios 503, debe pasar temporalmente a estado
abierto:
CLOSED -> OPEN -> HALF_OPEN -> CLOSED
Ejemplo:

   •   429: respetar Retry-After.
   •   Si no existe Retry-After, aplique enfriamiento de 30 a 120 segundos.
   •   Durante ese período, enrutar directamente a ZAI.
   •   Probar OpenAI con una sola petición en estado HALF_OPEN.
   •   Si vuelve a fallar, amplíe el enfriamiento.

Además:

   •   Limitar la concurrencia por proveedor.
   •   Usar un depósito de tokens por RPM/TPM.
   •   Aplicar retroceso exponencial con jitter .
   •   Evitar reintentar errores de formato como si fueran indisponibilidad.
   •   Mantener estadísticas por proveedor, modelo y rol.

Ejemplo de métricas adaptativas:
provider_success_rate provider_429_rate provider_invalid_json_rate
provider_p95_latency provider_cost_per_valid_idea
El enrutador debería optimizar el coste/latencia por resultado válido , no solo intentar
proveedores en un orden fijo.

1.4 Conmutación por error demasiado lenta

Ejemplo:
13:26:39 provider_unavailable_failover 13:28:33 failover_succeeded
Son aproximadamente 114 segundos para completar el proveedor alternativo.
Otro caso:
13:34:26 provider_unavailable_failover 13:36:10 failover_succeeded
Aproximadamente 104 segundos .
Esto puede deberse a:

   •   Tiempo de espera excesivo.
   •   Generaciones con demasiados tokens.
   •   Cola interna del proveedor.
   •   Reintentos dentro del adaptador antes de la conmutación por error.
   •   Falta de límites de tiempo por fase.

Mejora recomendada

Definir presupuestos explícitos:
connect_timeout: 5 s first_token_timeout: 15–30 s
total_request_timeout: 60–90 s max_provider_attempts: 1 o 2
max_total_operation_time: 90–120 s
Para una API de streaming, también emite eventos periódicos:
{ "type": "provider_waiting", "provider": "zai", "elapsed_seconds": 30
}
Si el costo lo permite, se puede usar una petición especulativa o solicitud cubierta
cuando el proveedor primario supere el percentil esperado. Debe aplicarse con cuidado
para no duplicar costes.


2. Problemas de ejecución del motor evolutivo
2.1 Espacio QD sobredimensionado para la población

La rejilla es:
grid_shape=(10, 10, 8)
Eso produce 800 celdas , pero la población es de solo 8 individuos y hay 3
generaciones.
No necesariamente es un error, pero la ocupación máxima será extremadamente baja.
Con tan pocas evaluaciones, el archivo MAP-Elites/Quality-Diversity puede ser
demasiado disperso para aportar valor.

Opciones

    •   Reduzca inicialmente la rejilla, por ejemplo 5 × 5 × 4 = 100.
    •   Usar contenedores adaptativos o CVT-MAP-Elites.
    •   Incrementar el presupuesto de evaluaciones.
    •   Separar population_sizede offspring_per_generation.
    •   Registrador ocupación y mejora real del archivo:

archive_coverage qd_score new_cells_filled cells_improved
elite_replacement_rate behavior_descriptor_distribution
Una configuración debe dimensionarse por el número total esperado de evaluaciones, no
solo por la dimensionalidad conceptual.

2.2 Semántica poco clara ensurprise_gate_skipped
surprise_gate_skipped skipped=2 threshold=0.1 total_saved=2
No queda claro si:

    •   Se omitió el filtro y se guardaron dos ideas.
    •   Dos ideas fueron descartadas por el filtro.
    •   Se saltaron dos evaluaciones.
    •   total_saved=2es acumulado o de esa operación.


Mejora recomendada

Usar eventos y campos inequívocos:
surprise_gate_evaluated candidates=5 surprise_gate_rejected count=2
threshold=0.1 surprise_gate_accepted count=3 ideas_persisted count=3
total_run_persisted=12
También conviene registrar la distribución:
surprise_score_min surprise_score_mean surprise_score_max
Un umbral fijo de 0.1podría ser demasiado bajo o alto dependiendo de cómo se
normalice la métrica. Sería preferible calibrarlo con percentiles o datos históricos.

2.3 Falta un cierre explícito de ejecución

No aparece:
evolution_completed
ni:
evolution_failed
Esto dificulta saber si el proceso:

      •   Terminó correctamente.
      •   Sigue ejecutándose.
      •   Fue cancelado por el cliente.
      •   Quedó bloqueado.
      •   Se interrumpió al cerrarse el stream.

Mejora recomendada

Toda ejecución debe terminar en un estado final persistente:
CREATED RUNNING COMPLETED PARTIAL FAILED CANCELLED TIMED_OUT
Evento final sugerido:
evolution_completed run_id=... duration_ms=... generations_requested=3
generations_completed=3 ideas_generated=... ideas_valid=...
ideas_archived=... provider_failovers=... estimated_cost=...
También deberían existir puntos de control por generación para poder reanudar una
ejecución.


3. Carga tardía del encoder
El codificador se descarga/carga durante la primera evolución:
Warning: unauthenticated requests to the HF Hub Loading weights...
encoder_loaded model=all-MiniLM-L6-v2
Esto añade latencia al primer usuario y crea una dependencia de red en tiempo de
ejecución.

Mejora recomendada

      •   Descargue el modelo durante la construcción de la imagen.
      •   Mantenerlo en caché persistente.
      •   Precargarlo durante el arranque o antes de declarar el servicio ready.
      •   Configurar HF_TOKENsi se necesita acceder al Hub.
      •   Fijar revisión/hash del modelo para reproducibilidad.
      •   Comprobar memoria si habrá varios trabajadores: cada proceso podría
          cargar su propia copia.

/health/live /health/ready

disponibles.


4. Observabilidad y registros
4.1 Registros INFO aparecen como errores

Estas líneas están clasificadas como [err]:
INFO: Started server process INFO: Application startup complete INFO:
Uvicorn running...
Uvicorn escribe parte de sus logs en stderr, por lo que la plataforma los etiqueta como
errores aunque no lo sean.

Mejora

Configurar logging.config.dictConfigpara:

   •   Enviar INFO a stdout.
   •   Enviar ERROR a stderr.
   •   Usar JSON en producción.
   •   Eliminar secuencias ANSI.

Actualmente aparecen códigos como:
[32m[1m
Esto ensucia la indexación y las búsquedas.

4.2 Retraso entre timestamp interno y timestamp del colector

Ejemplo:
timestamp exterior: 13:24:58 timestamp del evento: 13:24:48
Hay retrasos de varios segundos entre la emisión y la captura. Puede deberse un
almacenamiento en búfer.

Mejora

   •   Desactivar el buffering de Python: PYTHONUNBUFFERED=1.
   •   Usar un controlador que haga fluir.
   •   Emitir marcas de tiempo UTC estructuradas.
   •   Medir log_ingestion_delay_ms.
   •   Evitar barras de progreso de Hugging Face/TQDM en producción.
4.3 Falta de contexto uniforme

run_idaparece en evolution_started, pero no en mutaciones, cruces, fallos de
proveedor o generación de población.
Esto complica correlacionar eventos cuando hay varias ejecuciones simultáneas.

Campos que deben acompañar casi todos los logs

run_id request_id generation operation_id agent provider model attempt
duration_ms input_tokens output_tokens cost
Para mutaciones y cruces, incluya también generation.
Además, el campo challengecontiene parte del contenido del usuario. Aunque está
truncado, podría filtrar información sensible. Es preferible registrador:
challenge_hash challenge_length domain
y dejar el contenido completo fuera de los logs por defecto.

URL de documentación

Transmisión

El POST /api/v1/evolution/streamregresa 200al inicio, pero eso no implica éxito de
la evolución. El protocolo debería emitir:

   •   run_started
   •   generation_started
   •   generation_progress
   •   provider_failover
   •   checkpoint_saved
   •   run_completedorun_failed
   •   latidos del corazón

También debe cancelar el trabajo si el cliente se desconecta, salvo que la ejecución esté
configurada como persistente en el segundo plano.
6. Plan de mejoras priorizado
P0 — Corrección y estabilidad

   1. Normalizar las respuestas de cada proveedor.
   2. Usar salida estructurada con JSON Schema/Pydantic.
   3. Corregir la cardinalidad: nunca aceptar más de lo pedido y rellenar
      faltantes.
   4. Disyuntor de implementación párr 429/503.
   5. Añadir tiempos de espera por proveedor y presupuesto total por operación.
   6. Persistir un estado final de cada ejecución.
   7. Propagar run_idy generationa todos los logs.

P1 — Rendimiento

   1. Precargar all-MiniLM-L6-v2.
   2. Reducir reintentos inútiles contra proveedores limitados.
   3. Controlar concurrencia y cuotas por RPM/TPM.
   4. Hacer lotes más pequeños y reparables.
   5. Ajustar el tamaño de la rejilla QD al presupuesto de evaluaciones.
   6. Añadir puntos de control para reanudar ejecuciones.

P2 — Observabilidad y calidad

   1. Registros JSON en ANSI.
   2. Métricas de latencia, coste, análisis y conmutación por error.
   3. Eventos de streaming de progreso.
   4. Métricas QD: cobertura, puntuación QD y celdas mejoradas.
   5. Alertas por:

   •   población incompleta;
   •   tasa de JSON inválido;
   •   429;
   •   ejecución sin evento final;
   •   duración excesiva.
